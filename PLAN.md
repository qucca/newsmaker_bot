# PLAN.md — разбивка MVP на цепочку задач

План построения MVP по границам и порядку из `CLAUDE.md` и `docs/design.md`.
Принцип: по одной задаче за раз, без скаффолдинга наперёд; решения по логике —
за пользователем (согласуем перед реализацией пунктов, помеченных ⚠️).

Порядок: сначала фундамент → полный цикл **только на L1** → **Google News отдельным
шагом**. Глобальный проход (юзеро-независимый) строим раньше персонального — его
можно проверять прямо по БД, ещё до онбординга.

---

## Фаза 0 — фундамент

- [x] **T1. Bootstrap и тулчейн.** package.json (`engines` Node 24.x), tsconfig strict,
  ESLint+Prettier, npm-скрипты, `.gitignore`, `.env.example`, пустой каркас `src/`.
  Без бизнес-логики. *(детали ниже)*
- [x] **T2. Config (`src/config`).** Загрузка+валидация env через zod (токен бота,
  Anthropic key, путь к БД, кап юзеров и т.п.), fail-fast, секреты не в логах.
- [x] **T3. DB-слой (`src/db`).** Один инстанс `better-sqlite3`, раннер миграций
  (`migrations/*.sql` по порядку), первая миграция со схемой MVP. ⚠️ Точный DDL
  (таблицы/поля/индексы) — решение по логике, согласуем перед написанием. Здесь же
  добавляется рабочий `npm run migrate`.

## Фаза 1 — рабочий цикл на L1 (ядро)

- [x] **T4. Сбор L1 (`src/sources`).** Чтение списка фидов из `sources`, фетч через
  `rss-parser`, изоляция дохлых фидов, таймауты+retry/backoff, conditional GET
  (ETag/If-Modified-Since), кап кандидатов. (Google News тут НЕТ.)
- [x] **T5. Канонизация URL + дедуп прогона.** Канонизация → `articles.canonical_url`
  UNIQUE, дедуп ДО обогащения. Детерминированно → юнит-тесты. (Резолв обёрток не
  нужен — L1 даёт чистые URL.)
- [x] **T6. LLM-клиент (`src/llm`).** Провайдеро-независимая обёртка (Anthropic/OpenAI/Google
  + fake), structured output + zod-валидация + один ретрай, рычаг кеша промпта; Batch —
  зарезервированный тип. Выбор провайдера/моделей через env (`LLM_PROVIDER`, `LLM_MODEL_*`).
- [x] **T7. Глобальное обогащение (`src/enrich`).** Один батч-вызов Claude → сущности/теги,
  язык-независимый ключ, качество, флаг срочности, **нейтральные факты**.
  ⚠️ Схема ответа + промпт — согласуем.
- [x] **T8. Кросс-язычная кластеризация (`src/cluster`).** Ключ = нормализованный набор
  топ-сущностей, точное совпадение в скользящем окне ~48–72ч; трекинг
  `first_seen`/`updated_at`/счётчик источников/представитель. Детерминированное ядро
  → тесты. ⚠️ Окно и правила тай-брейка — согласуем.
- [x] **T9. Онбординг + команды (`src/bot`).** `/start` (язык → интересы-кнопки → опц.
  текст → окна+tz → объём → мини-подборка), `/settings`, `/delete`, `/block`,
  предохранитель (кап/allowlist + рейт-лимит). Создаёт юзеров.
  ⚠️ Тексты/кнопки/категории — согласуем. ⚠️ Механизм предохранителя (кап / allowlist / оба)
  выбираем здесь — отложено из D4; allowlist-таблицу заводим, только если выберем allowlist.
- [x] **T10. Ранжирование per-user (`src/score`).** Отсев `sent_log` (по `cluster_id`)/блоков,
  скор = пересечение тегов − штраф за источники, топ-N. Без LLM. Детерминированно → тесты.
- [x] **T11. Рендер саммари (`src/render`).** Per-`(кластер, язык)`, Claude пишет
  заголовок+нейтральное саммари из канонических фактов, кеш `summaries`,
  переиспользование. ⚠️ Промпт/схема — согласуем.
- [x] **T12. Сборка карточки.** Кеш-рендер + ссылка на представителя + строка «почему ты
  это видишь», управление `disable_web_page_preview`. ⚠️ Формат карточки — согласуем.
- [x] **T13. Очередь отправки (`src/send`).** Троттлинг (~1/с на чат, ~30/с глобально),
  retry/backoff, 429→`retry_after`, 403→`active=0`, `sent_log` сразу после ack,
  уступать event loop.
- [ ] **T14. Кнопки 👍/👎 (`src/bot`).** Компактный `callback_data` id, запись фидбэка как
  агрегируемых сигналов (по источнику/тегу).
- [x] **T15. Планировщик (`src/scheduler`).** Один тикер, расчёт окон по tz + `last_sent`,
  состояние в БД, вычисление разведено с отправкой.

## За пределами MVP (не сейчас)

LLM-ранжирование, эмбеддинги в кластеризации, авто-предложение блока, keyword-фиды,

## Технический долг — ретенция БД (выявлено при дизайне T8, не потерять; вне MVP-цикла)

`articles` копит строки на ВСЕ прогоны, `clusters` — на все окна; без чистки БД растёт
неограниченно. Нужна периодическая ретенция — отдельный maintenance-тик или хук
планировщика T15 (НЕ cron-на-юзера). Что и как чистить (горизонты/частота/триггер —
решение по логике, согласовать перед реализацией):

- **articles**: статьи старше окна кластеризации + запас (за окном новые статьи к старому
  кластеру уже не приходят). Учесть FK `cluster_id ON DELETE SET NULL` и порядок удаления.
- **clusters**: вышедшие из окна И не нужные для ранжирования/отправки; `summaries` уедут
  каскадом (`ON DELETE CASCADE`).
- **sent_log**: ⚠️ это и есть дедуп отправки по `cluster_id`. Горизонт ретенции `sent_log`
  ОБЯЗАН быть НЕ короче максимального возраста, при котором кластер ещё может быть выбран к
  отправке (T10) — иначе всплывёт повторная отправка. Чистить позже самих кластеров.
- **feedback / blocked_sources**: по возрасту НЕ чистить (агрегируемые сигналы обучения;
  `cluster_id ON DELETE SET NULL` сохраняет голос/издание при чистке кластера).

## Решения T14 (кнопки 👍/👎 + запись фидбэка) — согласовано

- **Сигнал — только `source`** (издание представителя кластера). Штраф по источнику уже
  работает в T10 (`selectSourcePenalties`). Тег-агрегация НЕ делается (нет потребителя;
  `cluster_id ON DELETE SET NULL` сделал бы тег-снимок хрупким — денормализацию тегов, если
  понадобится, делать вместе с ретенцией).
- **Голос на уровне кластера** (`feedback.cluster_id`): одна карточка = одна история;
  пишем источник представителя — ровно то, что юзер видел. НЕ размазываем по всем источникам кластера.
- **UX:** тап → тост + отметка выбранной кнопки (`👍 ✓` / `👎 ✓`); переголос меняет голос
  (UPSERT по `(chat_id, cluster_id)`); тап по той же кнопке = no-op (клавиатуру не редактируем,
  избегаем «message not modified»). Нет источника представителя → честный тост `fb_stale`, без записи.
- **Гейт «калибровка»:** кнопки вешаются только на первых **N=30** карточках, что юзер вообще
  получил (`COUNT(*) FROM sent_log WHERE chat_id` < N в момент сборки карточки). Потом фид чистый.
  Решение принимается ПРИ ОТПРАВКЕ, старые сообщения НЕ редактируем → не нужен трекинг `message_id`
  и миграция. N=30 — порог-решение пользователя; вынести в config (проброс — T15).
- **callback_data:** префикс `fb~` (рядом с `ob~`/`set~`/`del~`), `fb~up~<clusterId>` / `fb~down~<clusterId>`.

### Follow-up из T14 (вне текущего среза)

- **Постоянный канал фидбэка после калибровки:** тоггл в настройках «кнопки on/off» ИЛИ режим
  `/rate` (пере-показ последних историй из `sent_log` + кеш `summaries` с кнопками). Сейчас после
  калибровки канала обратной связи нет — осознанно (фидбэк — обучающий сигнал, не ядро доставки).
- **Реакции вместо кнопок (пост-MVP):** чище фид, но требует трекинга `message_id ↔ cluster_id`
  (апдейт `message_reaction` не несёт `cluster_id`) + маппинг произвольных эмодзи в ±1. ПЕРЕД
  реализацией проверить доставку `message_reaction` в ЛС (allowed_updates; поведение для приватных чатов).
- **Снятие кнопок со старых сообщений** (если захочется ретро-чистки) — нужен `message_id` + проход
  `editMessageReplyMarkup`; намеренно не делаем (поздний голос — валидный сигнал).
- **T15:** проброс `CALIBRATION_CARDS` (=30) из config в `buildUserCards` через `deps`; маппинг
  `CardMessage.replyMarkup → reply_markup` и `disableWebPagePreview → link_preview_options` в реальном `deps.send`.
- **Калибровка vs ретенция `sent_log`:** счётчик калибровки читает `COUNT(sent_log)`. Когда появится
  ретенция `sent_log`, счёт сбросится и кнопки вернутся — тогда перейти на отдельный lifetime-счётчик
  или базу по `created_at`.

## Решения T15 (планировщик) — согласовано

- **Окна = локальные времена-точки** `reading_windows` (`08:00/13:00/19:00/22:00`), не диапазоны.
  `latestDueWindow(windows, tz, now)` (luxon) даёт **последнюю границу, наступившую сегодня** в локальном
  дне юзера; `null` — если сегодня ещё ни одна не наступила. **Догон только самого свежего окна и только
  в пределах локального дня** (вчерашнее после даунтайма не досылаем).
- **Due** = `last_sent_at IS NULL OR last_sent_at < граница`. Обработали → `setLastSent(chatId, граница)`
  (двигаем на границу, не на now). Один дайджест за тик на юзера. НЕ cron-на-юзера; состояние — только
  `users.last_sent_at` (переживает рестарты), отдельной таблицы нет.
- **Пустой дайджест / сбой обработки → окно всё равно закрываем** (двигаем `last_sent_at`): одно окно =
  одна попытка. Заглушку «новостей нет» НЕ шлём.
- **Рендер on-demand в окне** (Batch отложен): `processUser` = `scoreForUser` → `renderPairs` (кеш по
  `(cluster,lang)`) → `buildUserCards` → `sendUserCards`. **Глобальный проход** (`collect`+`persist`→
  `enrich`→`cluster`) — на **отдельном таймере** `GLOBAL_PASS_INTERVAL_MIN` (дефолт 30), идёт ДО окон;
  тик окон — `TICK_INTERVAL_MIN` (дефолт 15). Оба под `guardedRunner` (защита от наложения), глоб.проход
  прогревается один раз при старте. Шаги глоб.прохода изолированы (сбой одного не блокирует остальные).
- **Проводка (`src/index.ts`)**: реальный send-адаптер `CardMessage → bot.api.sendMessage`
  (`replyMarkup→reply_markup`, `disableWebPagePreview→link_preview_options`); один LLM-клиент на обе роли.
  **Fail-soft**: нет LLM-конфига → планировщик не стартует, бот живёт для онбординга/команд.
- **Проброс config через `deps`** (T7/T11/T14 follow-up закрыты): `MAX_ENRICH_BATCH`/`ENRICH_RUN_CAP`,
  `CLUSTER_WINDOW_HOURS`/`CLUSTER_RUN_CAP`, `SCORE_WINDOW_HOURS`, `RENDER_MAX_OUTPUT_TOKENS` (новый дефолт-слот
  в `RenderDeps`), `CALIBRATION_CARDS`. Дубль-дефолты внутри модулей оставлены как mirror (тестируемость).

### Follow-up из T15 (вне текущего среза)

- **Ретенция БД** (тех.долг выше) — отдельный maintenance-тик/хук планировщика; согласовать горизонты.
- **`writeEnrichment AND enriched_at IS NULL`** (follow-up T7) не понадобился: `guardedRunner` гарантирует

- **Batch API / eager-прогрев кеша** саммари до окна — пост-MVP (заглушка `BatchCapable`).
- **«Первый дайджест в следующее окно»**: новый юзер (`last_sent_at IS NULL`) ловит уже прошедшую сегодня

- `startScheduler` (таймеры/композиция) юнит-тестом не покрыт — покрыта логика (`latestDueWindow`,

## Решения T16 (Google News L2) — согласовано

- **Скоуп (зафиксирован docs):** только `gnews_topic` — страновые top-stories по `hl/gl/ceid` +
  тематические разделы. `gnews_search` (точечный keyword под юзера) остаётся ТОЛЬКО в схеме, не
  реализуется (design откладывает точечные keyword-поиски за пределы MVP).
- **Стратегия раскрутки = вариант A (свой резолвер за интерфейсом, без сторонней либы-декодера).**
  Живой зонд (2026) показал: base64-декод обёртки мёртв, HTTP-редирект уводит обратно в consent-флоу
  GN (не к издателю). Единственный рабочий путь — **batchexecute-handshake**: GET страницы обёртки
  (consent-куки + браузерный UA) → выдрать `data-n-a-id`/`data-n-a-sg`/`data-n-a-ts` → POST на
  `/_/DotsSplashUi/data/batchexecute` (rpcid `Fbv4je`, payload `garturlreq`) → распарсить реальный URL
  из ответа (`)]}'`-префикс → `wrb.fr`/`Fbv4je` → `garturlres`). Резолвер за интерфейсом
  `resolve(link)→Promise<string|null>` — заменяемо при смене формата. ToS-серая зона (batchexecute +
  consent-куки) принята осознанно.
- **Изоляция:** нерезолвнутый кандидат (`null`) выкидывается, прогон не падает; любой сбой шага → `null`.
  L1 обязан давать ценность при полностью выключенном GN.
- **Порядок пайплайна:** новый async-шаг `resolveCandidates` МЕЖДУ `collect` и `persist` (канонизация/
  дедуп синхронные) — design «резолв → дедуп → обогащение». Триггер раскрутки — по хосту `news.google.com`;
  не-GN кандидаты проходят насквозь без изменений.
- **`source` после раскрутки** = хост раскрученного URL (как в L1, `publisherFromUrl`), фолбэк — исходный
  `source`. GN `<source>`-тег не используем.
- **Резолв последовательный + уступание event loop**, пер-айтемный таймаут + retry/backoff
  (переиспользуем `retry.ts`). Параллелизм/кеш раскрутки — follow-up.
- **kill-switch `GOOGLE_NEWS_ENABLED`** (дефолт `false`): GN-источники сидятся всегда, но включаются в
  прогон только при `true` (чтение фидов гейтится флагом). БД при выключении не трогаем.
- **Билдер URL `src/sources/gnews-url.ts`** (чистая функция): `topic='TOP'` → `/rss?hl..gl..ceid..`;
  раздел → `/rss/headlines/section/topic/<TOPIC>?…`. Имя фида деривируется. `feeds.gn.ts` — зеркало
  `feeds.l1.ts`: пустой список + пример, реальный маппинг язык→hl/gl/ceid = контент пользователя.
- **Сидинг** обобщён на GN (`kind='gnews_topic'`) рядом с L1; upsert по `url`.

### Follow-up из T16 (вне текущего среза)

- **Кеш раскрутки** wrapped→resolved: избежать повторного резолва (стоимость сети) при дрейфе
  conditional-GET; ограниченный параллелизм резолва вместо строго последовательного.
- **Хрупкость фикстур:** фикстуры HTML-обёртки и ответа batchexecute протухнут при смене формата GN —
  переснять зондом и поправить парсер. Покрытие — изоляцией (`null`), не падением прогона.
- **Детект тихой поломки резолва (слепое пятно):** при смене формата GN резолв ломается МОЛЧА —
  изоляция отдаёт `null`, L1 не страдает, а юнит-тесты остаются зелёными (гоняются против застывшей
  фикстуры, не против живого Google). Узнать можно только по факту «GN-кандидатов стало ноль». Нужен
  детект: метрика «доля раскрученных GN за прогон» + алерт по массовому логу `gn candidate dropped:
  unresolved`; опц. живая канарейка (раскрутить эталонную ссылку на старте/по расписанию, не валя прогон).
  Обязательно ДО включения GN в проде (`GOOGLE_NEWS_ENABLED=true`).
- **`gnews_search`** (keyword-фиды, в т.ч. per-user под интересы) — отдельная итерация поверх рабочего GN.
- **consent-куки / таймаут резолва** вынести в config, если понадобится тюнинг.

## T1 (Bootstrap) — что конкретно войдёт

Чисто механический тулчейн, без бизнес-логики и без продуктовых решений.

**Войдёт:**
- `package.json`: `engines.node` = 24.x, `type: module`, npm как менеджер; скрипты
  `dev` / `build` / `start` / `lint` / `format` (рабочие сразу). `migrate` добавим
  в T3, когда появится раннер.
- `tsconfig.json`: TypeScript strict (без `any`).
- ESLint + Prettier: конфиги в корне, проходит линт.
- `.gitignore` (вкл. `.env`, `node_modules`, `dist`, `*.sqlite`).
- `.env.example`: пустые значения под ключи (`TELEGRAM_BOT_TOKEN`, `ANTHROPIC_API_KEY`,
  путь к БД и т.п.) — без реальных секретов.
- Пустой каркас каталогов по структуре из `CLAUDE.md`
  (`src/bot|sources|enrich|cluster|score|render|send|scheduler|db|llm|config`,
  `docs/`, `migrations/`).
- Минимальный `src/index.ts` (точка запуска-заглушка: бизнес-логика снаружи неё,
  чтобы переход на webhook был заменой одной точки).

**НЕ войдёт (отложено в следующие задачи):** загрузка/валидация env (T2),
подключение к БД и миграции (T3), любая логика пайплайна и бота.

**Зависимости в `package.json`:** только нужное для T1 (TS, eslint, prettier).
`grammY`, `better-sqlite3`, `rss-parser`, Anthropic SDK, `luxon`, `zod`,
`node-cron`/`node-schedule` — по мере появления задач, не наперёд.

**Контент для T1 не нужен.** Понадобится в фазе 1: список языков (+ маппинг
`hl`/`gl`/`ceid`), категории интересов, стартовый список L1-RSS. Секреты (токен бота,
Anthropic key) — для запуска, не для написания кода.

## Отложено: контент L1 (фиды / языки / категории)

Наполнение `src/sources/feeds.l1.ts` реальным кураторским списком L1-RSS **отложено**
(решение пользователя). Код T4 от этого не блокируется: работает на фикстурах, сидинг
пустого списка = no-op, `npm run seed:sources` и тесты проходят.

**Что нужно будет дать** (контент, решает пользователь):
- список поддерживаемых **языков** (ISO 639-1);
- кураторский **список L1-RSS**: `name / url(именно фид) / lang / categories`;
- словарь **categories** — обязан совпасть с категориями интересов онбординга (T9),
  иначе не сойдётся маршрутизация «фид → интерес юзера».

**Когда становится необходимо:**
- **Мягко (первая реальная польза): перед T7–T8.** Обогащение (T7) и кластеризацию (T8)
  осмысленно гонять по живым кандидатам, а не фикстурам; T7 ещё и платный (Claude).
  До этого T5–T8 строятся и юнит-тестируются без реального списка.
- **Словарь categories — на T9** (онбординг), вместе с категориями интересов.
- **Жёстко (блокер): до T13** (очередь отправки) и любой реальной рассылки дайджеста —
  без заполненных `sources` слать нечего.
