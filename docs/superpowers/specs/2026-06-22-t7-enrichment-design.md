# T7 — Глобальное обогащение (`src/enrich`) — спецификация

Дата: 2026-06-22. Фаза T7 из `PLAN.md`. Решения по логике согласованы с пользователем
(см. раздел «Согласованные решения»). Контекст «почему» — `docs/design.md`, конвенции — `CLAUDE.md`.

---

## 1. Назначение и границы

Один проход на запуск (юзеро-независимый, общий для всех). Берёт **новых** кандидатов
(`articles.enriched_at IS NULL`), батч-вызовом LLM извлекает нейтральные факты, сущности,
теги, оценку качества и флаги срочности, и **записывает результат обратно в `articles`**.

- Кластеры (`clusters`) T7 **не трогает** — их наполняет T8, «промотируя» обогащение
  от представителя кластера.
- Срочный контур и рассылка — вне T7 (`is_urgent`/`is_major` считаются, но по ним
  ничего не отправляется в MVP).
- Это **синхронный** батч-вызов LLM (`generateStructured`: массив на входе → массив
  на выходе), **не** Anthropic Batch API. `BatchCapable` остаётся зарезервированным
  швом (T11/T15), как зафиксировано в T6.

## 2. Согласованные решения (логика продукта)

1. **Вход LLM = заголовок + RSS-описание.** Сохраняем `contentSnippet`/`description`
   из фида и подаём в LLM (богаче нейтральные факты, осмысленный quality). Требует
   мини-миграции (новая колонка) и правки парсера T4.
2. **Теги — из фиксированного словаря.** Единый управляемый словарь категорий
   (общий контракт T7→T9→T10). LLM выбирает теги **только** из него (через zod-`enum`).
   Наполнение провизорное (12 slug'ов), финализируется на T9; **механизм** — сейчас.
3. **`neutral_facts` — на языке оригинала статьи.** Render (T11) переводит на язык юзера.
4. **`quality` — LLM оценивает содержательность 0–100** (реальная новость vs
   кликбейт/реклама/листикл/мнение). Для L1 — мягкий сигнал; переиспользуется на GN (T16).
5. **Сущности и `cluster_key` — язык-независимые** (канонические английские имена,
   где есть общепринятые), несмотря на язык фактов. Иначе кросс-язычная кластеризация
   T8 не склеит «одну историю из разных языков».
6. **Без фактчек-прохода / детекции «фейка»** — whitelist-only (кураторский L1).

## 3. Изменение схемы — миграция `0002`

```sql
-- migrations/0002_articles_description.sql
ALTER TABLE articles ADD COLUMN description TEXT;  -- RSS-сниппет, вход обогащения; NULL если фид не дал
```

- STRICT-таблица допускает `ADD COLUMN` (nullable, без DEFAULT).
- `description` хранит **усечённый** plain-text сниппет (кап ~1000 символов).
- Полный текст оригинальной статьи **не** тянем и не храним (копирайт, вне MVP).

## 4. Правка T4 — сбор RSS (`src/sources`)

- `parse.ts`: в `ParsedItem` и `toCandidate` добавить `description`:
  `item.contentSnippet ?? stripHtml(item.content)`, усечь до капа. `null`, если фид
  ничего не дал.
- `types.ts` (`RawCandidate`), `persist.ts`, `ArticleInsert` и `INSERT_ARTICLE`

- Существующие тесты `parse.test.ts` / `persist.test.ts` дополнить фикстурой с описанием
  и кейсом «фид без описания» (`null`).

## 5. Контракт LLM (zod, structured output)

Один объект на статью; на входе — массив статей. Привязка выход→статья по локальному
индексу `ref`.

```
CategorySlug = z.enum(CATEGORIES)   // см. §6

EnrichItem = {
  ref: number,                 // = индекс статьи во входном батче (целостность маппинга)
  entities: string[],          // 1..6, канонические (англ. имена где есть), по убыванию значимости
  tags: CategorySlug[],        // 0..4, ТОЛЬКО из словаря — off-словарный тег = провал схемы
  quality: number,             // int 0..100, рубрика содержательности
  is_urgent: boolean,          // breaking / время-чувствительное (в MVP не рассылается)
  is_major: boolean,           // крупное мировое событие
  neutral_facts: string[]      // 2..6 коротких нейтральных утверждений НА ЯЗЫКЕ ОРИГИНАЛА
}

Response = EnrichItem[]
```

- `cluster_key` LLM **не** возвращает — выводится детерминированно в коде (§6).
- Валидация: если количество элементов или набор `ref` не совпадает со входом — это
  провал схемы (zod + проверка целостности) → один ретрай (в клиенте) → при повторном
  провале чанк пропускается (§7).
- Никакого парсинга «почти-JSON» регэкспами — только structured output + zod.

### Промпт

- **system-блок** (`PromptBlock.cache:true`): описание задачи, словарь категорий,
  рубрики `quality`/`is_urgent`/`is_major`, требование нейтральности фактов и правило
  «факты на языке оригинала», явный запрет тегов вне словаря, требование вернуть `ref`.
- **input-блок**: батч статей — `ref`, `source`, `lang`, `title`, `description`.
  Если `lang` у статьи `NULL` (фид не указал) — LLM определяет язык по
  `title`/`description` и пишет `neutral_facts` на нём.
- При записи `is_urgent`/`is_major` (boolean из LLM) конвертируются в `0/1`
  под CHECK-констрейнты схемы.

## 6. Детерминированные модули (юнит-тесты с фикстурами)

- **`src/categories.ts`** — единый источник истины словаря (общий для enrich/bot/score):
  массив slug'ов + `CategorySlug` (`z.enum`). Провизорный набор (12):
  `world, politics, business, technology, science, health, sports, culture,
  entertainment, environment, society, security`. Расположен в корне `src/`,
  т.к. контракт сквозной (T7/T9/T10), а не привязан к одной стадии.
- **`src/enrich/cluster-key.ts`** — `deriveClusterKey(entities): string`:
  взять топ-K (K=5) сущностей → нормализация (lowercase, снять диакритику/пунктуацию,
  схлопнуть пробелы) → dedup → сортировка → склейка через `|`. Пустой вход → пустой
  ключ (статья без сущностей не кластеризуется по ключу). T8 переиспользует модуль.
- **усечение описания** (`stripHtml` + кап) и **маппинг `ref`→`articleId`** и
  **чанкинг** — детерминированы → отдельные тесты.

## 7. Поток и устойчивость (`src/enrich/index.ts`)

`enrichPending(db, llm, cfg)`:

1. `resolveLlmConfig` на старте → fail-fast при отсутствии ключа провайдера
   (follow-up из T6: сейчас ключ сделан `.optional()`).
2. Выбрать необогащённых через partial-индекс `idx_articles_unenriched`; кап на прогон.
3. Разбить на чанки `MAX_ENRICH_BATCH` (дефолт ~20) — бюджет токенов.
4. Каждый чанк → `generateStructured` (zod-валидация + 1 ретрай внутри клиента).
   Провал после ретрая → **структурный лог + пропуск чанка**: статьи остаются
   `enriched_at IS NULL` и дообработаются в следующий прогон. Один битый чанк
   не валит весь проход (изоляция сбоя).
5. На успехе — в транзакции записать поля обогащения + `enriched_at` + `cluster_key`
   для всех статей чанка.
6. Структурный лог итогов: обогащено / пропущено / usage. **Без** заголовков,
   контента, персональных данных и секретов в логах.

`src/db/articles.ts`: добавить `selectUnenriched(db, limit)` и `writeEnrichment(db, rows)`
(batched-транзакция, синхронный better-sqlite3).

## 8. Конфиг (`src/config`)

- `MAX_ENRICH_BATCH` (дефолт ~20) и кап кандидатов на прогон — через env с дефолтами.
- Кап усечения описания — константа (или env, дефолт ~1000).

## 9. Тестирование

- Стек: `node:test` + `tsx`, colocated `*.test.ts`, исключены из сборки.
- Юнит-тесты с фикстурами: `deriveClusterKey` (нормализация, топ-K, dedup, сортировка,
  пустой вход), усечение/`stripHtml`, чанкинг, маппинг `ref`→`articleId`, zod-схема
  (валидный/невалидный/off-словарный тег/рассинхрон `ref`).
- Оркестрация `enrichPending` — через **fake-провайдер** LLM (уже есть в `src/llm`):
  выборка→чанкинг→запись, изоляция битого чанка, идемпотентность (повторный прогон
  не трогает уже обогащённые), запись `description` из T4.

## 10. Решения реализации (по умолчанию, согласованы)

- `K=5` сущностей для `cluster_key`.
- `MAX_ENRICH_BATCH ≈ 20`.
- Усечение описания ~1000 символов.
- `src/categories.ts` в корне `src/`.
- Стратегия изоляции сбоя — пропуск чанка (не падение прогона).
