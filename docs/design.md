# Telegram-бот — персональный новостной куратор

## Что это

Многопользовательский Telegram-бот, который несколько раз в день присылает каждому пользователю в личные сообщения подборку новостей под его интересы. Каждая новость — это карточка: заголовок, краткое саммари **на языке пользователя**, ссылка на оригинал и кнопки 👍/👎. Источники многоязычные; заголовок и саммари приводятся к выбранному языку. Подбор делает LLM, дубликаты отсекаются, одна и та же история из разных изданий схлопывается в одну.

Ключевая ценность — не «много новостей», а **релевантность и отсутствие дублей**. Именно на этих двух вещах проект живёт или умирает, поэтому они вынесены в ядро пайплайна. Отсюда же следует, что **единица контента — это кластер («история»), а не отдельная статья**: дедуп, отправка и обучение работают на уровне кластера.

---

## Стек (Node.js / JavaScript)

- **grammY** — бот-фреймворк
- **rss-parser** — сбор RSS
- **провайдеро-независимый LLM-слой** (`src/llm`, выбор через `LLM_PROVIDER`) — обогащение,
  ранжирование (позже) и написание саммари; Anthropic — рекомендуемый провайдер по умолчанию,
  но взаимозаменяем с OpenAI / Google
- **better-sqlite3** — данные (синхронный; тяжёлые циклы держим асинхронными/троттлёными вокруг него)
- **node-cron / node-schedule** — тикер расписания
- декодер завёрнутых URL Google News (проверить актуальность пакета перед вшиванием)

> **Резолв URL Google News — самая хрупкая зависимость всей системы.** Google периодически меняет кодировку обёрток, и библиотеки-декодеры ломаются; местами раскрутка требует запроса к их batchexecute-эндпоинту, а не простого декода. Поэтому: нужен фолбэк (HTTP-фетч с проходом по редиректам как план Б), изоляция ошибки (нерезолвнутый кандидат выкидывается, а не валит прогон), и принцип «**Уровень 1 (прямые RSS) должен давать ценность сам по себе**» — Google News не делаем критической зависимостью MVP.

Отложено за пределы MVP: **Telegraph** (карточка-сообщение уже содержит то же, что и страница — отдельный хоп и копирайт-риск без выгоды) и **Pexels** (стоковое фото на конкретной новости вводит в заблуждение).

## Архитектура пайплайна — два уровня

Стоимость и сложность падают, если развести дорогой общий проход и дешёвый персональный. Расходы растут как `O(кандидаты) + O(кластеры × языки)`, а не `O(юзеры × все кандидаты)` и даже не `O(юзеры × шорт-лист)`.

### Глобальный проход (один раз за запуск, общий для всех)

1. Широкий сбор кандидатов из RSS (см. раздел «Источники»).
2. Резолв канонических URL — Google News заворачивает ссылки в свой редирект, без раскрутки дедуп по URL ломается на каждой такой новости.
3. **Дедуп кандидатов внутри прогона по каноническому URL** — один и тот же URL приходит и из L1, и из Google News; иначе платим за обогащение дважды. Порядок жёсткий: резолв → дедуп → обогащение.
4. **Один батч-вызов Claude** обогащает каждого нового кандидата: сущности/теги, нормализованный язык-независимый ключ, оценка качества, оценка срочности + флаг «крупное мировое событие», **нейтральные факты в каноническом виде** (это вход для рендера саммари, НЕ финальный текст карточки). Поле «ключевое слово для картинки» в MVP не считаем — картинки отложены.
5. **Кросс-язычная кластеризация** «одна история из разных изданий» — **входит в MVP**, потому что именно она даёт заявленную ценность «нет дублей». Поскольку источники на разных языках, кластеризация идёт по сущностям/нормализованному ключу, а не по строкам заголовков (эмбеддинги — позже, если ключа будет не хватать). Кластер-ключ MVP = нормализованный набор топ-сущностей (lowercase, отсортировать, склеить), точное совпадение = один кластер. У каждого кластера трекаем `first_seen`, `updated_at`, число независимых источников и представителя для ссылки (по качеству источника, тай-брейк — статья на языке юзера если есть в кластере).

> **Временное окно кластера (важно).** Матчинг по ключу делается только в скользящем окне ~48–72ч (по `first_seen`/`updated_at`). Иначе старый кластер начнёт ловить свежие новости с теми же сущностями и склеивать события разных дней (вчерашний и сегодняшний матч одной команды — разные истории). За окном — новый кластер, даже при совпадении ключа.

### Проход по юзеру (по шорт-листу, дёшево)

1. Отсев уже отправленного (`sent_log`, **по `cluster_id`**) и заблокированных источников.
2. **Ранжирование — per-user, без LLM (в MVP):** скор = пересечение тегов профиля с тегами кластера − штраф за дизлайкнутые/заблокированные источники. Отсортировали, взяли топ-N. LLM-ранжирование добавляем позже, если эвристика начнёт мазать (та же логика «эмбеддинги потом»).
3. **Считаем множество нужных пар `(кластер × язык)`** = объединение шорт-листов всех юзеров с их языками. 500 русскоязычных читателей одной истории дают одну пару `(cluster, ru)`, а не 500.
4. **Рендер саммари — по парам `(кластер, язык)`, с кешем:** для каждой ещё не отрендеренной пары один вызов Claude пишет заголовок + **нейтральное** саммари на целевом языке из канонических фактов кластера. Результат кладём в кеш `summaries(cluster_id, lang)` и переиспользуем для всех юзеров этого языка. Переводим только своё саммари, не полный текст оригинала — копирайт чистый, ссылка ведёт на оригинал.
5. **Сборка карточки:** готовый рендер из кеша + ссылка на представителя кластера + строка **«почему ты это видишь»** из пересечения тегов (без LLM).

> **Ранжирование и рендер саммари — разные вызовы с разной кардинальностью.** Ранжирование per-user (в MVP — арифметика, 0 вызовов LLM). Рендер — per-(кластер, язык), кешируется. Сливать их в один per-user-вызов нельзя: это и есть та переплата, при которой идентичное саммари пишется заново на каждого юзера.

> **Саммари нейтральное, не персонализированное.** Текст карточки отвечает на «что произошло» — он одинаков для всех на языке X. Для новостного продукта нейтральность — это фича: персонализированное «саммари под твой интерес» начинает подкручивать акценты под профиль. Ощущение «это про меня» даёт строка «почему ты это видишь», а не сам текст новости.

> Единственный LLM-расход, растущий с аудиторией, — рендер саммари, и он растёт по числу **языков**, а не юзеров. Глобальный проход фиксирован, ранжирование в MVP без LLM.

## Источники — три уровня

**Уровень 1 — прямые RSS изданий (фундамент).** Самый чистый источник: настоящий заголовок и URL, часто медиа-поля с картинкой, самое быстрое обновление. Кураторский список — по горстке надёжных изданий на каждый поддерживаемый язык. **L1 обязан давать ценность даже при полностью отключённом Google News.**

**Уровень 2 — Google News (catch-all).** Широкая страховочная сеть поверх первого уровня; бесключевой, по предсказуемым URL-шаблонам. Используем три типа фидов: топ-стори по стране/языку (`hl`/`gl`/`ceid` — это же рычаг под нужный язык), тематические, и поиск по ключевому слову под интересы. Оговорки: ссылки завёрнуты (нужен резолв с фолбэком, см. «Стек»), качество источника не гарантировано (поэтому оценка качества обязательна в глобальном проходе), «For You/Following» в RSS недоступны.

**Уровень 3 — нишевые/тематические агрегаторы.** По желанию и позже, под специфические интересы.

Для MVP: общий пул уровня 1 + несколько широких тематических/страновых Google-фидов на язык. Точечные keyword-поиски под конкретного юзера добавляем позже, когда станет видно, каких тем системно не хватает.

## Картинки

По умолчанию карточка **без картинки**. Стоковые фото к конкретным новостям не подмешиваем — случайное фото вводит в заблуждение, это хуже, чем без картинки. Превью оригинала отдаём через стандартный предпросмотр ссылки Telegram (поведение превью непредсказуемо — нет og:image, сайт блокирует превью; протестировать на реальных источниках и управлять `disable_web_page_preview`). Pexels — максимум для мягких/вечнозелёных тем, если вообще.

## Отправка

Очередь с троттлингом и retry/backoff под лимиты Telegram (≈1 сообщение/сек на чат, ~30/сек глобально; массовая рассылка иначе молча теряется на 429). Цикл рассылки — асинхронный и троттлёный вокруг синхронного SQLite, и должен **уступать event loop**, чтобы команды юзеров оставались отзывчивыми во время большой рассылки.

- **429** — ждать ровно `parameters.retry_after` из ответа Telegram, не общий backoff.
- **403 (бот заблокирован юзером)** — ловить, помечать `users.active = 0`, прекращать слать; иначе очередь долбится в мёртвые чаты.
- **Идемпотентность:** `sent_log` пишется по каждому сообщению **сразу после ack** Telegram (ключ `(chat_id, cluster_id)`), чтобы краш посреди рассылки докатывался без дублей.

## Расписание

Один тикер каждые ~15–30 мин. Для каждого юзера на тике считаем, «попадает ли сейчас в его окно чтения», из его таймзоны и `last_sent`. Никаких cron-задач на каждого пользователя. Состояние живёт в БД → устойчиво к рестартам. 3–4 окна в день под пики чтения, ограниченное число новостей за раз.

- **Развести вычисление и отправку.** Глобальный проход и рендер дайджеста к окну не привязаны — их можно гонять с запасом времени *до* окна и наполнять кеш (тогда применим Batch API). В момент окна — только отправка из готового. Так батч не конфликтует с окнами чтения.
- **Таймзонную математику не катать руками** (DST, юзеры около полуночи) — брать `luxon` / `date-fns-tz` / `Temporal`.

## Онбординг и многопользовательность

Прогрессивный онбординг по `/start` — кнопки до первой ценности, длинный текст опционально (иначе люди отваливаются на «опишите ваши интересы»):

1. Короткое приветствие.
2. **Язык** — предзаполняем из `language_code` клиента, подтверждение кнопкой. Это же язык саммари и рычаг для Google News.
3. **Интересы — кнопки (мульти-выбор)** по крупным категориям. Дают структурированные теги и **маршрутизируют источники**: именно они определяют, какие фиды и Google-News-запросы тянуть под юзера.
4. **Уточнение текстом (опционально)** — любимые темы, люди, компании. Идёт в свободный текстовый профиль, даёт LLM нюанс. Пропуск разрешён.
5. **Окна чтения + таймзона** — пресетами кнопок; смещение выводим из страны/языка, точную tz уточняем только если не вывелось. Геолокацию не просим.
6. **Объём** — сколько новостей за раз и сколько раз в день, пара кнопок.
7. **Подтверждение + немедленная мини-подборка** из 1–2 новостей — момент «оно работает», не заставляем ждать планового запуска.

Теги, свободный текст и сигналы фидбэка храним **раздельно**. Всё редактируется через `/settings`. Команда `/delete` удаляет данные юзера.

> **Бот публичный** — любой может нажать `/start`, и каждый юзер добавляет линейный расход. На MVP заложить предохранитель: кап на число юзеров или allowlist + рейт-лимит на онбординг.

## Обучение

Лайки/дизлайки копятся как **структурированные сигналы** и подмешиваются в промпт скоринга (когда он станет LLM-ным) **рядом** с исходным профилем, не переписывая его (защита от дрейфа — модель могла бы тихо выкинуть интерес). Исходный текст интересов юзера остаётся неизменным.

> Фидбэк подаётся **агрегированным и ограниченным**, а не сырым логом событий: свёртка по источнику и по тегу (напр. «источник N: +3/−12», «тег crypto: −8»). Сырой растущий список раздувал бы промпт, размывал сигнал и постоянно инвалидировал кеш профиля. В MVP, где ранжирование без LLM, та же агрегация даёт штраф за источники прямо в арифметике скора.

## Отсев изданий — два механизма

- Явная команда `/block <издание>`.
- Авто-предложение скрыть источник после порога дизлайков — **не режем молча, спрашиваем**.

## Срочные новости (v1.5 — отдельная подсистема)

Не часть планового дайджеста, а **второй, более частый контур** (тикает ~раз в 5 мин), opt-in. Полное per-user ранжирование не делает — только сверку breaking-кластеров с важными тегами и глобальным тумблером.

**Детект срочного — два сигнала, которые должны сойтись вместе:**
- **Скорость роста кластера** — история за короткое окно всплыла у многих независимых изданий (`first_seen` + счётчик источников).
- **Классификация в глобальном проходе** — LLM пометил как срочное / крупное событие (поля уже считаются там, маргинальная стоимость ~0).

Одиночный «breaking» от одного издания не шлём — для срочного приоритет точности над полнотой.

**Два типа:**
- *Персональное срочное* — по темам, отмеченным юзером как важные/любимые.
- *Глобальные триггеры* (война, катаклизмы) — отдельный тумблер, **обязательно opt-in**.

**Анти-спам (без него фича убивает доверие):**
- Рейт-лимит на юзера (N в день + кулдаун).
- Дедуп против дайджеста в обе стороны (тот же `cluster_id`).
- Тихие часы через таймзону, два тира: «важное» ждёт окна бодрствования, «критическое» (крупная катастрофа) шлётся немедленно невзирая на час.

> Развивающаяся история (в кластер дотекают новые статьи) здесь же даёт «обновление по истории X» вместо новой карточки — это сценарий именно срочного контура, в дайджесте кластер один раз показан и в `sent_log`.

## Данные (SQLite)

| Таблица | Содержимое |
|---|---|
| `users` | chat_id, таймзона, язык, теги интересов, свободный профиль, окна чтения, объём, `active` (0 при 403), настройки срочных (opt-in темы, тумблер глобальных событий, тихие часы/тир) |
| `articles` | кандидаты после резолва: `canonical_url` (UNIQUE, основа дедупа в прогоне), источник, язык, исходный заголовок, `cluster_id`, время фетча |
| `clusters` | кластеры историй: язык-независимый ключ, **нейтральные факты** (вход для рендера), теги, сущности, качество, флаг срочности, `rep_article_id` (представитель для ссылки), счётчик независимых источников, `first_seen`, `updated_at` (для скользящего окна матчинга ~48–72ч) |
| `summaries` | **кеш саммари**, ключ `(cluster_id, lang)`: заголовок и нейтральное саммари на языке `lang`. Одна строка переиспользуется всеми юзерами этого языка |
| `feedback` | лайки/дизлайки как структурированные сигналы, с `source` для агрегации; читаются свёрткой, не сырьём |
| `blocked_sources` | блок-лист по юзеру |
| `sent_log` | что отправлено кому, **ключ `(chat_id, cluster_id)`** (дедуп по кластеру, не по URL), с флагом срочное/дайджест; пишется после ack, с ретенцией |
| `sources` | фиды уровня 1 + шаблоны Google-News-запросов с привязкой к языку/категории |

Дедуп ведётся отдельно по каждому юзеру, **по `cluster_id`**.

## Модель и стоимость

**Haiku 4.5 (Anthropic) — дефолт-рекомендация.** Это её профиль: классификация, извлечение, короткая суммаризация. **Sonnet 4.6** — точечно и только если на каком-то языке не хватит гладкости саммари (эскалируем только шаг рендера саммари, роль `render`). Opus не нужен. LLM-слой провайдеро-независим: Anthropic — один из взаимозаменяемых провайдеров (наравне с OpenAI и Google), выбор провайдера и моделей — через env (`LLM_PROVIDER`, `LLM_MODEL_DEFAULT`/`LLM_MODEL_RENDER`); рекомендация по умолчанию остаётся Haiku-класс.

**Где живёт стоимость:**
- Глобальный проход (обогащение + кластеризация) — **фиксированный**, ~$5–10/мес на всю систему (с батчем вдвое меньше), размазывается на всех.
- Ранжирование per-user — в MVP **0 LLM** (арифметика по тегам).
- Рендер саммари — единственный расход, растущий с использованием, но по числу `(кластеры × языки)`, **а не юзеров**: 500 русскоязычных = одно русское саммари на кластер.

**Рычаги стоимости:** Batch API (−50%; дайджест не латентно-чувствителен и считается с запасом до окна — идеален под батч; срочный контур латентный, батч там нельзя, но объём мал), кеширование промпта (−90% на повторяющемся входе — кешируем системную инструкцию и канонические факты), кеш `summaries` (полное переиспользование на язык), матчинг и ранжирование без LLM.

**Порядок цифр:** при аудитории от ~30–50 человек итог — **примерно $0.30–1.00 на активного юзера в месяц** (ниже прежней вилки за счёт переиспользования саммари на язык и ранжирования без LLM); вилку двигают число запусков в день, число языков и Batch/кеширование.

## Хостинг

Небольшой always-on процесс (дешёвый VPS / Raspberry Pi / бот-хостинг): бот одновременно слушает юзеров в реальном времени и шлёт по расписанию — чистый GitHub Actions не подходит. **API дешевле своего GPU**, пока аудитория меньше ~100–150 постоянных юзеров (свой always-on GPU простаивает 90%+ времени при всплесковой нагрузке, плюс хуже многоязычное качество и devops-время). Промежуточный вариант на потом — та же open-weight модель через чужой inference-провайдер, как эксперимент после MVP со слепым сравнением саммари.

## Границы MVP

**Отгружаем сначала:**
RSS → резолв/дедуп кандидатов → глобальное обогащение → **кросс-язычная кластеризация** → отсев `sent_log`/блоков → ранжирование по тегам (без LLM) → рендер саммари на язык юзера (кеш `summaries`) → карточка со ссылкой на оригинал + «почему ты это видишь» → 👍/👎.

**Порядок внутри MVP (важно):** первый рабочий цикл строим **только на L1** (прямые RSS изданий) — чистые URL, резолв не нужен, ядро видно сразу. **Google News прикручиваем вторым шагом**, отдельным коммитом, когда цикл на L1 уже работает: так не блокируемся на самой хрупкой части (резолв обёрток) и сразу проверяем главный вопрос — попадает ли подбор в интересы.

**Наращиваем поверх работающего цикла:**
LLM-ранжирование (если тегов мало) → эмбеддинги в кластеризации (если ключа мало) → авто-предложение блока источника → точечные keyword-фиды под юзера → срочный контур → (при желании) картинки и Telegraph.

Так быстрее всего видно главное — попадает ли подбор в интересы. Это и есть единственный вопрос, определяющий, будут ли ботом пользоваться.
