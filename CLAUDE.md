# CLAUDE.md

Конвенции и правила работы над проектом для Claude Code. Это файл про **«как»**.
Контекст «что и почему» — в `docs/design.md`, читай его при вопросах о продуктовых решениях.

---

## Что это за проект

Многопользовательский Telegram-бот — персональный новостной куратор. Несколько раз
в день шлёт каждому пользователю в ЛС подборку новостей под его интересы: карточка
с заголовком, кратким саммари **на языке пользователя**, ссылкой на оригинал и
кнопками 👍/👎. Источники многоязычные, дубликаты отсекаются, одна история из разных
изданий схлопывается в одну. Подбор делает LLM (дефолт — модель класса Claude Haiku 4.5,
провайдер выбирается через конфиг).

**Единица контента — кластер («история»), а не статья.** Дедуп, отправка и обучение
работают на уровне `cluster_id`. Полное описание архитектуры и пайплайна — в `docs/design.md`.

**Статус: MVP-цикл построен и закоммичен** (sources → enrich → cluster → score → render →
card → send → scheduler, онбординг и 👍/👎). Контент засеян (L1-фиды), деплой/ретенция/алерты
готовы. Дальше — пост-MVP итерации поверх работающего цикла; состояние и дорожная карта — в `PLAN.md`.

## Технические решения (зафиксировано)

- **Язык:** TypeScript (strict). Никакого `any` без явного обоснования в комментарии.
- **Runtime:** Node.js LTS (на сейчас 24.x). Зафиксировать в `package.json` → `engines`.
- **Менеджер пакетов:** npm. Использовать только npm (`package-lock.json` коммитим).
- **Режим бота:** long polling (`bot.start()`). НЕ webhook.
  - Причина: маленький always-on процесс, не нужен публичный URL/TLS.
  - Код бота держать так, чтобы переход на webhook был заменой одной точки запуска,
    без изменения хендлеров и логики. Вся бизнес-логика — вне точки запуска.
- **Бот-фреймворк:** grammY.
- **БД:** SQLite через `better-sqlite3` (синхронный).
- **LLM:** провайдеро-независимый слой (`src/llm`). Провайдер и модели выбираются через
  env (`LLM_PROVIDER`, `LLM_MODEL_DEFAULT`/`LLM_MODEL_RENDER`); поддержаны Anthropic / OpenAI /
  Google. Дефолт — модель класса Haiku, эскалация рендера — через роль `render`.
- **Расписание:** node-cron / node-schedule — один тикер, НЕ задача на юзера.
- **Таймзоны:** через `luxon` / `date-fns-tz` / `Temporal`. Руками арифметику tz/DST НЕ катать.

## Структура проекта

Модули по стадиям пайплайна. Не сваливать всё в один файл.

```
src/
  bot/          # grammY: хендлеры команд, онбординг, кнопки фидбэка
  sources/      # сбор RSS: фиды уровня 1 + Google News (резолв завёрнутых URL) + feeds.l1.ts/feeds.gn.ts + сидинг
  enrich/       # глобальный проход: батч-обогащение кандидатов через LLM (нейтральные факты)
  cluster/      # кросс-язычная кластеризация: ключ из сущностей, скользящее окно ~72ч
  score/        # проход по юзеру: ранжирование тегами БЕЗ LLM (штраф за источники)
  render/       # рендер саммари на (кластер × язык) с кешем — отдельно от ранжирования
  card/         # детерминированная сборка HTML-карточки (без БД/IO)
  send/         # очередь отправки с троттлингом и retry/backoff
  scheduler/    # тикер: расчёт окон чтения по таймзоне + last_sent; вычисление отдельно от отправки
  db/           # better-sqlite3: подключение, миграции, репозитории, ретенция
  llm/          # провайдеро-независимая обёртка (Anthropic/OpenAI/Google), structured output + валидация
  notify/       # observability: алерты админу в Telegram + детект стрика сбоев
  log/          # структурное логирование
  config/       # загрузка и валидация env
  categories.ts # единый словарь категорий (enrich-enum + интересы онбординга + ранжирование)
  langs.ts      # поддерживаемые языки (ru/en)
  index.ts      # точка запуска (composition root) — бизнес-логика снаружи
docs/
  design.md     # полное описание продукта (источник истины по «почему»)
migrations/     # .sql файлы миграций по порядку
PLAN.md         # состояние проекта + дорожная карта (что готово, решения, открытые follow-up)
.env.example    # шаблон секретов с пустыми значениями
Dockerfile / docker-compose.yml / docker-entrypoint.sh  # деплой (long polling, named volume)
```

## Правила разработки

### Решения по логике — спрашивать ВСЕГДА

- **Решения по логике и продуктовому поведению принимает пользователь, не Claude.**
  Перед любым выбором, который меняет *как ведёт себя продукт* или *как устроен
  пайплайн*, — остановись и спроси, не реализуй на своё усмотрение.
- Спрашивать обязательно, если:
  - выбор не зафиксирован явно в этом файле или в `docs/design.md`;
  - есть несколько разумных вариантов, и они дают разное поведение
    (структура данных, формат карточки/сообщений, пороги, тексты юзеру,
    что считать дублем, логика ранжирования/кластеризации/расписания,
    схема промптов и схемы ответов LLM, новые поля/таблицы);
  - задача тянет за собой то, чего нет в текущем срезе MVP;
  - всплыло противоречие между задачей и доками.
- Если выбор уже однозначно зафиксирован в доках — следуй ему, не переспрашивай.
- Чисто механические детали реализации (имена переменных, мелкий рефакторинг,
  раскладка кода внутри согласованной структуры) решай сам.
- Формат вопроса: коротко изложи варианты и трейд-оффы, дай свою рекомендацию,
  **дождись ответа** — и только потом реализуй. Не принимай решение «по умолчанию».
- При сомнении «спросить или нет» — спрашивай.

### Дисциплина скоупа — ВАЖНО

- **MVP-цикл уже построен** (полный пайплайн RSS → … → 👍/👎 + кластеризация). Дальнейшую
  работу ведём тем же принципом: **по одному срезу за раз**, не скаффолдим наперёд,
  останавливаемся и сверяемся. Состояние и дорожная карта — в `PLAN.md`.
- **Кластеризация — ядро ценности «нет дублей».** Ключ = нормализованный набор топ-сущностей,
  матчинг **только в скользящем окне ~72ч** (иначе старый кластер склеит события разных дней).
- **Google News (L2) — за kill-switch** `GOOGLE_NEWS_ENABLED` (дефолт `false`); L1 самодостаточен.
  Резолв обёрток GN — самая хрупкая зависимость, не блокируемся на ней.
- Пост-MVP (итерации **поверх** работающего цикла, см. `PLAN.md`): LLM-ранжирование, эмбеддинги
  в кластеризации, entity-following, срочный контур, авто-предложение блока, точечные keyword-фиды,
  Batch API, картинки, Telegraph.
- Если задача тянет за собой то, чего нет в текущем срезе, — остановись
  и спроси, а не реализуй наперёд.

### LLM-вызовы

- Только через structured output / tool use. **Никакого парсинга «почти-JSON»
  регэкспами.** Описывай ожидаемую схему ответа явно + валидируй (zod) и делай
  один ретрай при провале схемы.
- Все вызовы LLM идут через единый клиент-обёртку в `src/llm/` (со structured output +
  zod-валидацией + одним ретраем при провале схемы), независимо от выбранного провайдера.
- Два прохода с **разной кардинальностью** (детали в `docs/design.md`):
  - глобальный проход — один батч-вызов на запуск, общий для всех; обогащение
    выдаёт **нейтральные факты** (вход для рендера), не финальный текст карточки;
  - проход по юзеру разнесён на два разных шага:
    - **ранжирование — per-user, БЕЗ LLM в MVP** (пересечение тегов − штраф за источники);
    - **рендер саммари — per-`(кластер, язык)`, с кешем** `summaries(cluster_id, lang)`,
      переиспользуется всеми юзерами языка. НЕ per-user.
- **Ранжирование и рендер — разные вызовы.** НЕ сливать в один per-user-вызов
  (иначе идентичное саммари пишется заново на каждого юзера).
- **Саммари нейтральное**, не персонализированное под профиль (нейтральность — фича
  новостного продукта). «Это про меня» даёт строка «почему», а не текст новости.
- Закладывать рычаги стоимости: Batch API для планового дайджеста (считаем с запасом
  до окна), кеширование промпта для системной инструкции и канонических фактов, кеш `summaries`.
- Матчинг и ранжирование под профиль делать **без LLM** — пересечением тегов.

### База данных

- Один инстанс `better-sqlite3` на процесс.
- Миграции — обычные `.sql` файлы в `migrations/`, применяются по порядку.
- `better-sqlite3` синхронный: НЕ блокировать event loop тяжёлыми циклами.
  Цикл рассылки делать асинхронным/троттлёным вокруг синхронных запросов и
  **уступать event loop**, чтобы команды юзеров не подвисали во время рассылки.
- Дедуп кандидатов внутри прогона — по `articles.canonical_url` (UNIQUE),
  ПОСЛЕ резолва URL и ДО обогащения.
- **Дедуп отправки — по `cluster_id`, не по URL.** `sent_log` ключ `(chat_id, cluster_id)`.
  Это и есть «нет дублей»: тот же кластер из нового издания повторно не уходит.
- Кеш саммари — таблица `summaries` с ключом `(cluster_id, lang)`.
- Хранить раздельно: теги интересов, свободный текстовый профиль и сигналы
  фидбэка. Профиль пользователя **не переписывать** (защита от дрейфа).
- Фидбэк подаётся **агрегированным** (свёртка по источнику/тегу), не сырым логом
  событий — иначе раздувает промпт и инвалидирует кеш.

### Сеть и устойчивость

- Таймауты + retry/backoff на ВСЕХ внешних вызовах: RSS-фетч, резолв URL,
  LLM API (Anthropic/OpenAI/Google), Telegram API.
- Резолв URL Google News — с фолбэком (HTTP-редиректы как план Б) и изоляцией:
  нерезолвнутый кандидат выкидывается, не валит прогон. L1-фиды должны работать
  и при полностью отключённом Google News.
- Изоляция дохлых фидов (один упавший фид не валит прогон) + conditional GET
  (ETag / If-Modified-Since). Кап на число кандидатов за прогон.
- Отправка в Telegram — через очередь с троттлингом под лимиты
  (≈1 сообщение/сек на чат, ~30/сек глобально):
  - **429** — ждать ровно `parameters.retry_after`, не общий backoff;
  - **403** (бот заблокирован) — пометить `users.active = 0`, прекратить слать;
  - **идемпотентность** — `sent_log` писать по каждому сообщению сразу после ack.
- `callback_data` кнопок 👍/👎 — компактный id (лимит 64 байта), НЕ URL.
- Состояние расписания держать в БД, чтобы процесс переживал рестарты;
  вычисление дайджеста разнесено с отправкой (считаем до окна, шлём в окне).

### Секреты

- Все ключи — через `.env` + загрузка/валидация в `src/config/`.
- `.env` НИКОГДА не коммитим. Поддерживать `.env.example` с пустыми значениями.
- Секреты НИКОГДА не попадают в логи.
- Нужные ключи: токен бота (@BotFather), API-ключ выбранного LLM-провайдера
  (Anthropic / OpenAI / Google — см. `LLM_PROVIDER`).

### Качество кода

- Prettier + ESLint, конфиги в корне. Код проходит линт перед коммитом.
- Логирование структурное, без секретов и персональных данных в логах.
- Детерминированные куски (канонизация URL, дедуп, расчёт окна/таймзоны,
  троттл-очередь) — покрывать юнит-тестами с фикстурами.
- Осмысленные имена, маленькие функции, ошибки обрабатываем явно.

## Чего НЕ делать

- Не реализовывать фичи за пределами текущего среза MVP без явного запроса.
- Не принимать решения по логике/поведению продукта самостоятельно — спрашивать
  пользователя (см. «Решения по логике — спрашивать ВСЕГДА»).
- Не переписывать профиль пользователя силами LLM.
- Не писать саммари «под профиль» — оно нейтральное (искажение новости иначе).
- Не сливать ранжирование и рендер саммари в один вызов.
- Не дедупить отправку по URL — только по `cluster_id`.
- Не лить сырой лог фидбэка в промпт — только агрегированную свёртку.
- Не подмешивать стоковые картинки к конкретным новостям (вводит в заблуждение).
- Не копировать полный текст оригинальных статей (копирайт) — только своё саммари
  + ссылка на оригинал.
- Не парсить ответы LLM регэкспами.
- Не блокировать источники молча — только по `/block` или с предложением юзеру.
- Не плодить cron-задачи на каждого пользователя.
- Не катать арифметику таймзон руками.
